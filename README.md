> **Project closed 2026-05-04.** This repository is the empirical record
> of a 3-month research project on whether designing a deliberately
> AI-shaped programming language could let small local LLMs (≤22B
> parameters, consumer-GPU-hostable) do agentic-AI tooling work at
> cloud-quality reliability and reduced cost.

## TL;DR

| claim | result |
|---|---|
| Single-step tooling delegation to a fine-tuned local 7B-class ensemble reaches cloud parity | **Validated.** Stream C 29/30 (qwen-sigil-v7 + deepseek-sigil ensemble) vs Sonnet 4.6 29-30/30, ~6× cheaper per task, complementary failures (combined cascade 30/30 at <1% cloud cost). |
| Multi-step composition on the same local stack closes to cloud accuracy | **Refuted.** Best Sigil multi-step Path C: 7/30. The orchestration-ceiling diagnostic showed swapping only the per-step executor to Sonnet lifts the same chained pipeline to 26/30 — the gap is per-step executor capability, not chain design. |
| A novel syntactic surface (prefix-Lisp) is justified at consumer fine-tuning scale | **Refuted on cost grounds.** Same orchestration with qwen2.5-coder:7b writing Python (no fine-tune): 12/30 (+5 over Sigil at fixed 7B). Scaling the local Python executor to codestral:22b: still 12/30 (no scale benefit in this band). The 19-task gap to cloud decomposes ≈26% language-proximity / ≈74% scale-bound. |
| Pre-training a foundation model from scratch on a novel surface is feasible at consumer scale | **Refuted on hardware and corpus grounds.** Foundation pre-training requires trillions of tokens (we have 2,323) and ~$50K-5M of cloud GPU time (consumer hardware cannot reach this scale). |
| Effect-typed safety with capability-restricted IO would be valuable | **Sketched (mammouth U09, 2026-02), never built.** Identified as the strongest unbuilt thesis and the natural direction for follow-on work. |

## Reproducing benchmark numbers

Each result JSON records the model, prompts, hyper-parameters, seeds,
and per-task pass/fail. The exact commands:

- `benchmark/eval_real_tooling.py` — Stream C single-step suite
- `tools/agent_harness/agent_ab_harness.py` — A/B/C multi-step harness
  (with `--executor sigil|sonnet|python|python-ensemble` selector and
  `--cache-a` / `--cache-b` for not re-running deterministic baselines)
- `benchmark/finetune_local.py` — QLoRA training (Navi31-stable recipe)

All deterministic at temperature 0; retry attempts are the only
stochastic part and use a fixed temperature ramp.

## What remains valid for future research

These statements survive the project's close-out and are open
directions for follow-on work (detail in `papers/PAPER.md` §9):

1. **Capability-typed effects on a Python-subset host** (the unbuilt
   mammouth U09 thesis). The headline product positioning under this
   design — *"agents cannot delete production databases by
   construction"* — is concrete, demoable, and qualitatively stronger
   than "AI-native language."
2. **Capability-aware orchestrator-executor calibration as a research
   problem.** The orchestrator-quality regression measured (NH16
   showed Opus regressed Path C 7→4 at 6.6× cost) is the LLM analogue
   of the "teacher-student gap" in distillation; the cure is open.
3. **The "meeting halfway" methodology** generalizes beyond Sigil to
   any future fine-tuned-model-meets-novel-language design.
4. **Pre-training-language-proximity as a measurable axis**. A single
   data point (≈+5/30 at 7B from Sigil to Python) suggests the axis
   is real; systematic study across more languages, sizes, and task
   shapes is open.
5. **The privacy / confidentiality argument**
   (`papers/CONFIDENTIALITY_AND_LOCAL_LLMS.md`) is independent of the
   language-design subclaim and continues to motivate local execution
   regardless.

## Status of in-repo language artifacts

The Sigil language artifacts (`interpreter/`, `stdlib/`,
`LANGUAGE_SPEC.md`, `AGENTS.md`, `SIGIL-CORE.md`, `SIGIL-AGENT.md`,
`.sigil.grammar`) are preserved as the research artifact's
implementation. **They are not actively maintained.** Several of the
specification documents (LANGUAGE_SPEC.md, SIGIL-CORE.md,
SIGIL-AGENT.md) describe an early *two-layer* architecture
(Sigil-Agent surface compiling to a Sigil-Core IR) that did not
survive into the working language; the production interpreter is
single-layer S-expression with the meet-halfway aliases documented
in `papers/MEETING_HALFWAY.md`. Each spec file carries a banner
noting this.

## Methodology that ports forward

These techniques were validated empirically and survive any
language-design pivot:

- **Validator-in-loop** with surgical hints (Tier A static
  name-validity + Tier B principle-based 3B step-judge, both in
  `benchmark/`).
- **Cross-base ensemble** with fresh-prompt-temperature-0 on the
  alternative-base model (lifted Stream C from 26/30 to 29/30).
- **Sequential corpus generation** (~96% validity vs ~13% under
  parallel; details in
  `~/.claude/projects/.../memory/feedback_corpus_generation.md`).
- **Soft-pass scoring discipline** (NH9 confirmed the harness's
  exact-match grader is not deflating local accuracy; the gap to
  cloud is real, not byte-pedantry).
- **Meeting the model halfway** — when the model misuses a construct
  consistently, change the language layer to absorb the misuse rather
  than fighting it via prompt or training. (`papers/MEETING_HALFWAY.md`.)

These ports are independent of the specific language target.
